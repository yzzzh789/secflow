# 命名债与重复债

## 命名债

### 命名维度混用

当前模块命名混用了不同维度：

- 功能导向：`packet_analyzer`
- 设备导向：`nic_monitor`
- 工作流导向：`traffic_monitor`
- 泛化容器：`scripts`

结果是兄弟目录不在同一命名轴上，阅读时不容易建立稳定心智模型。

### 遗留目录命名歧义

- `traffic_monitor/` 这个名字看起来像当前主模块，但实际上更接近“遗留兼容区 + 仍被局部依赖的目录”。

### 目录名过宽

- `scripts/` 对当前项目来说过于宽泛，因为里面放的并不只是辅助脚本，而是当前产品主链路的一部分。

## 重复债

### Python 能力散落

仓库里有多份与流量、网卡、监控相关的 Python 入口脚本。哪些是当前主链路，哪些是历史实现，单看名字并不清楚。

### 前端与运行时的历史重叠

`static/` 是当前前端，但 `traffic_monitor/` 里仍然保留了旧版 HTML 和旧服务实现，这会提高误改错文件的概率。

### 文档重叠

当前文档和历史文档放在一起，如果没有统一索引，阅读者很容易把模块级历史说明误当成当前系统总说明。

## 当前最安全的治理方向

现在就适合做的：

- 写清楚哪些脚本是当前主链路
- 写清楚哪些目录属于 legacy
- 用统一的文档入口收敛整理判断

后续再做的：

- 按同一命名规则重命名模块
- 在有测试保障之后再抽公共 Python 层
- 在所有引用关系审计完后再删除或合并旧前端、旧运行时文件
